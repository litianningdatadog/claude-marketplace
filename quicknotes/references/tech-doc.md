# quicknotes — Technical Documentation

## Objective

In-context note-taking during a Claude Code session requires switching to an external tool or file, breaking flow. Conversely, asking Claude to "remember" something only persists within the session. `quicknotes` provides near-zero-friction, persistent note capture directly within Claude Code: any message prefixed with a capture intent immediately creates a dated, tagged, project-aware note stored as a Markdown file, queryable in future sessions.

---

## Problem Space

Claude Code memory (`~/.claude/projects/.../memory/MEMORY.md`) is designed for stable project facts, not ephemeral todos, reminders, or cross-project observations. Notes stored there tend to accumulate and require manual curation. A dedicated note store solves three friction points:

1. **Capture friction** — opening a note app mid-session breaks focus.
2. **Recall friction** — notes aren't surfaced proactively at session start or when switching to a relevant project.
3. **Lifecycle friction** — completing a task doesn't remove it from the list; stale notes accumulate.

---

## Solution Design

### Storage format

Each note is a single Markdown file under `~/.quicknotes/notes/` (or `$QUICKNOTES_HOME/notes/`). The filename is the note's ID (`YYYYMMDD-HHMMSS-<4hex>.md`). YAML-like frontmatter (parsed with stdlib, no third-party deps) stores structured metadata; the body follows after the closing `---`.

```
~/.quicknotes/notes/
└── 20260615-143022-a3f1.md
    ├── id: "20260615-143022-a3f1"
    ├── title: "first 80 chars of body"
    ├── created / updated: ISO-8601 UTC
    ├── priority: "normal"
    ├── project: "claude-marketplace"   ← detected from git remote
    ├── cwd: "/Users/tianning.li/..."
    ├── branch: "main"
    ├── tags: ["todo", "ux"]
    ├── due: null | ISO-8601 UTC
    └── refs: []                        ← cross-note links
```

### Path confinement

`write_note()` calls `_within(home, path)` to verify the resolved write target is under `~/.quicknotes` before creating or modifying any file. This prevents path traversal via crafted note content.

### Fuzzy resolution

Most management verbs accept an `id|fuzzy` argument. `resolve(home, ref)` first tries exact ID lookup, then scores all notes by term frequency (exact phrase matches score 3×, per-word hits score 1×). If exactly one note matches, it proceeds. If multiple match, it returns the candidates and asks the user to disambiguate — it never silently picks one.

## Features

### Capture

```bash
qn <note text> [#tag …] [--tag T]
```

- Inline `#hashtags` are extracted from the text (regex: `(?<![^\s])#([A-Za-z0-9_-]+)`), stripped from the stored body.
- Project and branch are auto-detected from `git remote get-url origin` and `git rev-parse --abbrev-ref HEAD`.
- Title defaults to the first 80 characters of the body.
- IDs are generated by the script; the agent never fabricates them.
- Due dates require a two-step flow: capture the note first (`qn <text>`), then set the due date separately (`qn update <id> --due <ISO>`). `cmd_capture` only parses `--tag`; passing `--due` at capture time treats it as note body text and leaves `due: null` in the frontmatter. The agent computes time expressions ("by Friday", "tomorrow") to ISO-8601 UTC before calling `qn update`.

### Management verbs

| Verb | Action |
|---|---|
| `list [--project P] [--tag T]` | List notes, optionally filtered |
| `search <query>` | Full-text search with TF scoring |
| `show <id\|fuzzy>` | Print full metadata + body |
| `done <id\|fuzzy>` | **Permanently delete** the note file |
| `update <id\|fuzzy> [flags] [body]` | Edit title, tags, priority, due date, or body |
| `due` | List past-due notes |
| `here` | List notes for the current project/cwd |
| `ref <id> <id>` | Create a bidirectional cross-note reference |

`done` is destructive and the SKILL.md notes this explicitly. `--tag` in `update` replaces the entire tag list (not appended).

### Proactive session reminders (opt-in)

An optional `SessionStart` hook runs `session_reminder.py`, which:
1. Calls `due_notes()` and `here_notes()`.
2. If either returns results, prints a single-line summary and up to 5 due note titles.
3. Always exits 0, wraps all logic in `except Exception: pass` — it must never break session start.

The SKILL.md provides the hook JSON inline and asks for user consent before installing.

### Shell alias installer (opt-in)

`install_alias.sh` appends a `qn()` shell function to `~/.zshrc`, `~/.bashrc`, or `~/.profile` (detected from `$SHELL`). Idempotent via a marker comment — re-running never double-installs. Requires confirmation unless `--yes` is passed.

## Architecture

```
quicknotes/
├── .claude-plugin/plugin.json              # Plugin manifest
├── commands/qn.md                          # /qn slash command (delegates to skill)
├── skills/quicknotes/SKILL.md             # Agent procedure + management commands
└── scripts/
    ├── notes_store.py                      # Core storage library (shared)
    ├── qn.py                               # CLI entrypoint (dispatches to notes_store)
    ├── session_reminder.py                 # SessionStart hook (due + here summary)
    └── install_alias.sh                    # Opt-in shell alias installer
```

### Module responsibilities

```
notes_store.py
├── default_home()          → $QUICKNOTES_HOME || ~/.quicknotes
├── new_id(now, rand)       → YYYYMMDD-HHMMSS-<4hex>
├── normalize_tag(tag)      → strips #, lowercase, whitespace→-
├── extract_hashtags(text)  → (cleaned_text, tags)
├── detect_project(cwd)     → git remote basename || dir name
├── detect_branch(cwd)      → git rev-parse --abbrev-ref HEAD
├── capture(text, ...)      → write new note, return note dict
├── get(home, nid)          → validated path read (ID safe regex + _within)
├── list_notes(...)         → reverse-chrono glob with optional filters
├── search(home, query)     → TF-scored full-text search
├── resolve(home, ref)      → exact-then-fuzzy, returns (note, candidates)
├── complete(home, nid)     → path.unlink() (hard delete)
├── update(home, nid, ...)  → read → patch _EDITABLE fields → rewrite
├── add_ref(home, a, b)     → bidirectional refs update on both notes
├── due_notes(home, now)    → ISO string comparison on due field
└── here_notes(home, ...)   → match by project name or exact cwd

qn.py
├── RESERVED verb set       → dispatches to cmd_* functions
├── cmd_capture             → falls through when no verb matched
└── _extract_opts(words)    → parses --key value pairs; --tag is multi-value
```

### No third-party dependencies

`notes_store.py` uses only stdlib: `json`, `pathlib`, `subprocess`, `datetime`, `re`. This ensures the script works in any Python 3.x environment without a `pip install` step.

## Design decisions

**Why flat files instead of a database?**
Flat Markdown files are inspectable, greppable, and git-friendly. The notes directory can be synced with any file sync tool. A database would require schema migrations and adds a runtime dependency.

**Why `done` deletes instead of archiving?**
Completed notes that accumulate create the same recall friction as incomplete ones. Deletion keeps the list signal-dense. Users who want archival can `show` before `done` or use `git` on a synced notes directory.

**Why ISO string comparison for `due_notes`?**
`YYYYMMDD-HHMMSS` ISO strings sort lexicographically identically to chronologically — a string `<=` comparison is correct and avoids datetime parsing overhead for the common case.
